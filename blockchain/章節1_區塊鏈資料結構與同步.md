## 1.1 區塊結構

### 區塊頭結構

區塊頭是區塊鏈中每個區塊的元數據部分，包含了驗證區塊完整性和維護鏈式結構所需的關鍵信息。

#### 核心組件

| 欄位 | 大小 | 用途 |
|------|------|------|
| 版本號 (Version) | 4 bytes | 標識使用的協議版本，支援分叉管理 |
| 前一區塊哈希 | 32 bytes (SHA-256) | 指向前一區塊的哈希值，建立鏈式結構 |
| Merkle Root | 32 bytes | 所有交易的 Merkle Tree 根哈希 |
| 時間戳 (Timestamp) | 4 bytes (Unix) | 區塊創建的大概時間 |
| 難度目標 (Difficulty Target) | 4 bytes (壓縮格式) | PoW 挖礦的難度閾值 |
| 隨機數 (Nonce) | 4 bytes | 挖礦過程中的可變參數 |

#### 設計原理
區塊頭的緊湊設計確保了：
1. **效率性**：輕節點只需下載區塊頭即可驗證
2. **安全性**：任何修改都會改變區塊哈希
3. **連接性**：前一區塊哈希建立不可變的鏈式結構

---

### 區塊體結構

區塊體包含了區塊的實際交易資料，是價值轉移和狀態變更的核心載體。

#### 交易結構組成

**輸入 (Inputs)**
- UTXO 引用：指向未花費輸出
- 解鎖腳本：證明對 UTXO 的所有權
- 序號：用於 RBF (Replace-By-Fee) 等功能

**輸出 (Outputs)**
- 金額：轉移的加密貨幣數量
- 鎖定腳本：定義花費條件
- 地址：接收方的地址

**其他欄位**
- 版本號：交易格式版本
- 鎖定時間：交易生效的最早時間

#### 設計考量
- **空間效率**：交易打包最大化，使用壓縮技術
- **驗證效率**：交易間的獨立性支援平行處理，透過 Merkle Tree 支援部分驗證

---

### 資料完整性保證

區塊鏈透過密碼學技術和結構設計確保資料的完整性。

#### 核心機制

**密碼學哈希**
- SHA-256：Bitcoin 使用的主要哈希算法
- 單向性：從哈希值無法反推原始資料
- 雪崩效應：微小變化導致完全不同的哈希值

**鏈式結構**
- 前向連結：每個區塊引用前一區塊的哈希
- 篡改檢測：任何歷史修改都會破壞後續區塊的連結
- 累積安全性：隨時間推移，修改成本指數級增長

**區塊驗證流程**
1. 結構驗證：檢查區塊格式正確性
2. 哈希驗證：確認前一區塊哈希匹配
3. Merkle 驗證：驗證交易列表完整性
4. 交易驗證：檢查每筆交易的有效性

## 1.2 Merkle Tree 建構與驗證

### 二元樹雜湊組合

Merkle Tree 採用二元樹結構，透過遞歸的雜湊組合方式，將大量交易資料壓縮成單一根哈希值。

#### 構建過程

**葉節點層**
- 每筆交易的哈希值作為葉節點：`Transaction Hash = SHA-256(Transaction Data)`
- 如果交易數為奇數，複製最後一個哈希

**內部節點層**
- 兩兩配對：`Parent Hash = SHA-256(Left Child + Right Child)`
- 遞歸構建直到只剩一個根節點

**根節點**
- Merkle Root：樹的頂端節點，代表所有交易的摘要
- 任何交易變化都會改變根哈希，記錄在區塊頭中

#### 演算法實現

```
function buildMerkleTree(transactions):
    hashes = [SHA-256(tx) for tx in transactions]
    
    while length(hashes) > 1:
        newLevel = []
        for i in range(0, length(hashes), 2):
            left = hashes[i]
            right = hashes[i+1] if i+1 < length(hashes) else left
            parent = SHA-256(left + right)
            newLevel.append(parent)
        hashes = newLevel
    
    return hashes[0]  // Merkle Root
```

---

### O(log n) 驗證複雜度

Merkle Tree 的對數複雜度特性使得驗證效率隨著交易數量呈對數級增長。

#### 複雜度分析

| 類型 | 複雜度 | 說明 |
|------|--------|------|
| 空間複雜度 | O(log n) | 需要存儲的兄弟節點數量 |
| 時間複雜度 | O(log n) | 需要執行的哈希運算次數 |
| 通信複雜度 | O(log n) | 需要傳輸的證明資料大小 |

#### 數據量對比

| 交易數量 | 線性驗證 | Merkle 驗證 | 效率提升 |
|----------|----------|-------------|----------|
| 1,000 | 1,000 | 10 | 100x |
| 1,000,000 | 1,000,000 | 20 | 50,000x |
| 1,000,000,000 | 1,000,000,000 | 30 | 33,333,333x |

#### 實際影響
- **Merkle Proof 大小**：約 32 bytes × log₂(n)
- **Bitcoin 實例**：2000 筆交易的區塊，證明大小約 352 bytes，對比完整區塊超過 1MB

---

### SPV 部分資料驗證

SPV (Simplified Payment Verification) 允許輕量級客戶端在不下載完整區塊鏈的情況下，驗證特定交易的存在性。

#### 核心原理
- **僅下載區塊頭**：包含 Merkle Root 的區塊頭鏈
- **按需查詢**：只在需要時請求特定交易的證明
- **Merkle Proof**：從目標交易到 Merkle Root 的哈希路徑

#### 驗證流程

```
function verifyMerkleProof(txHash, merkleProof, index):
    currentHash = txHash
    
    for siblingHash in merkleProof:
        if index % 2 == 0:  // 左子節點
            currentHash = SHA-256(currentHash + siblingHash)
        else:  // 右子節點
            currentHash = SHA-256(siblingHash + currentHash)
        index = index // 2
    
    return currentHash
```

#### 安全模型
- **信任假設**：假設網路中多數算力是誠實的，相信最長鏈
- **潛在風險**：隱瞞攻擊、網路分割、日蝕攻擊

#### 優化技術
- **Bloom Filter**：向全節點請求相關交易，模糊化實際關注的交易
- **檢查點**：預設歷史區塊為有效，從檢查點開始同步

## 1.3 UTXO 模型 vs 帳戶模型

### UTXO 未花費交易輸出

UTXO (Unspent Transaction Output) 模型是比特幣採用的核心帳本模型，將每筆資金視為離散的「未花費輸出」。

#### 交易流程

```
1. 選擇 UTXO
   錢包需要支付 0.5 BTC：
   - 找到一個或多個 UTXO 總額 ≥ 0.5 BTC

2. 構建交易
   輸入：UTXO_1: 1.0 BTC (從 Alice 錢包)
   輸出：
   - Output_1: 0.5 BTC (給 Bob)
   - Output_2: 0.4 BTC (找零給 Alice)
   - 手續費: 0.1 BTC (給礦工)

3. 更新 UTXO 集合
   移除：UTXO_1 (1.0 BTC)
   新增：Output_1 (0.5 BTC), Output_2 (0.4 BTC)
```

#### 設計優勢
- **並行處理**：不同 UTXO 可並行處理，無全域狀態衝突
- **隱私保護**：地址複用避免，找零機制提供自然的金額混淆
- **安全性**：UTXO 只能被花費一次，防止雙重花費

#### 實現挑戰
- **存儲效率**：大量小額 UTXO 增加存儲負擔
- **智能合約限制**：難以實現複雜的狀態機

#### 資料結構
```
UTXO = {
    txid: "交易哈希",
    vout: "輸出索引",
    amount: "金額",
    scriptPubKey: "鎖定腳本",
    height: "區塊高度"
}
```

---

### 帳戶模型餘額追蹤

帳戶模型是以太坊等區塊鏈採用的狀態模型，直接追蹤每個帳戶的餘額。

#### 帳戶類型
- **外部擁有帳戶 (EOA)**：由私鑰控制的用戶帳戶
- **合約帳戶**：由智能合約代碼控制的帳戶

#### 帳戶狀態
```
Account = {
    balance: "帳戶餘額",
    nonce: "交易計數器",
    storageRoot: "存儲樹根哈希",
    codeHash: "合約代碼哈希"
}
```

#### 設計優勢
- **程式設計簡潔**：直觀模型，類似傳統銀行帳戶
- **智能合約支援**：適合實現複雜的業務邏輯

#### 技術挑戰
- **併發處理**：多個交易同時修改同一帳戶的狀態衝突
- **存儲效率**：合約存儲無限增長，狀態爆炸風險

#### 狀態存儲實現
- **Merkle Patricia Tree**：前缀壓縮，高效存儲稀疏地址空間
- **快照機制**：定期創建狀態快照，支援快速同步

> **🔄 技術更新（2025+）**：Ethereum 正積極推進 **Verkle Tree** 以取代 Merkle Patricia Tree（EIP-6800）。Verkle Tree 使用向量承諾（基於橢圓曲線多項式），使 witness 大小從 ~MB 級降至 ~KB 級，是實現 **無狀態性（Statelessness）** 的關鍵：節點無需儲存完整狀態即可驗證區塊，預計在 2025-2026 年的 Ethereum 硬分叉中上線。

---

### 模型優缺點比較

| 特性 | UTXO 模型 | 帳戶模型 |
|------|-----------|----------|
| 程式設計複雜度 | 相對複雜 | 簡單直觀 |
| 隱私性 | 較強（地址多樣性） | 較弱（餘額透明） |
| 可擴展性 | 天然並行 | 受限於狀態衝突 |
| 智能合約 | 受限 | 友好 |
| 存儲效率 | UTXO 碎片化 | 狀態爆炸風險 |

#### 存儲規模（2024年實例）
- **Bitcoin UTXO 集合**：~4GB，約 1 億個 UTXO
- **Ethereum 世界狀態**：~100GB，~2 億個帳戶

> **🔄 技術更新（2025+）**：隨 Ethereum 鏈上活動持續成長，世界狀態規模到 2025 年已超過 200GB+，「狀態爆炸」問題更加嚴峻，這也推動了 **狀態過期（State Expiry / EIP-4444）** 提案——歷史超過一年的狀態將被歸檔，節點不再需要永久保存全部歷史。

#### 混合模型探索
- **Cardano Extended UTXO**：UTXO 可攜帶任意狀態資料，在 UTXO 基礎上實現合約功能
- **Fuel UTXO + VM**：可程式設計 UTXO，結合兩種模型優勢

## 1.4 區塊鏈資料庫儲存格式

### LevelDB 鍵值儲存

LevelDB 是 Google 開發的高性能嵌入式鍵值儲存引擎，被 Bitcoin Core 等項目採用。

#### 核心特性
- **LSM-Tree 架構**：寫入優化的資料結構，所有寫操作都是順序的
- **分層存儲**：資料按層級組織，新資料在上層
- **壓縮機制**：自動背景壓縮減少空間使用

#### Bitcoin Core 使用場景
```
chainstate/: UTXO 集合
- key: outpoint (txid + vout)
- value: coin (amount + script + height)

blocks/index/: 區塊索引
- key: block_hash
- value: block_metadata

blocks/: 原始區塊資料（LevelDB 存索引）
```

#### 讀寫流程
**寫入**：MemTable → WAL (日誌) → Immutable MemTable → SSTable（磁碟）

**讀取**：MemTable → Immutable MemTable → SSTable（按層級）→ Bloom Filter 快速判斷

#### 優勢與挑戰
| | 優勢 | 挑戰 |
|-|------|------|
| 性能 | 寫入性能優異 | 讀取放大（多層查找） |
| 空間 | 壓縮機制節省空間 | 壓縮過程臨時空間使用 |
| 可靠性 | WAL + 校驗和確保完整性 | 壓縮消耗 CPU 資源 |

---

### RocksDB 高效能引擎

RocksDB 是 Facebook 基於 LevelDB 開發的高性能嵌入式資料庫，被 Ethereum、Hyperledger 等廣泛採用。

#### 相對 LevelDB 的改進
- **多線程支援**：並行壓縮、並行刷新、讀寫並行
- **列族 (Column Families)**：邏輯上分離不同類型的資料
- **事務支援**：原子性的多鍵操作
- **備份與恢復**：內建的資料備份機制

#### Ethereum 使用場景
```
State Database (狀態資料庫):
- 帳戶狀態存儲
- 合約存儲資料
- Merkle Patricia Tree 節點

Chain Database (鏈資料庫):
- 區塊頭和區塊體
- 交易索引
- 收據資料
```

#### 效能優化配置
```
Block Cache: 熱資料緩存
Bloom Filter: 減少無效讀取
Compression: 減少存儲空間
Write Buffer: 批量寫入優化
```

#### 性能比較

| 特性 | RocksDB | LevelDB | 傳統 SQL |
|------|---------|---------|----------|
| 寫入性能 | 極高 | 高 | 中等 |
| 讀取性能 | 高 | 中等 | 高 |
| 功能豐富度 | 豐富 | 基本 | 非常豐富 |
| 運維複雜度 | 中等 | 低 | 高 |

## 1.5 資料同步

### 完整區塊鏈下載

Full Sync 從創世區塊開始下載並驗證整個區塊鏈的完整歷史，提供最高的安全性和去中心化程度。

#### 同步流程

```
1. 初始化階段
   - 連接到種子節點或已知節點
   - 請求區塊鏈頭部信息
   - 確定需要同步的區塊範圍
   - 建立本地資料庫結構

2. 區塊下載階段
for height in range(0, latest_height):
    block = download_block(height)
    verify_block(block)
    store_block(block)
    update_utxo_set(block.transactions)

3. 驗證流程
   - 區塊結構驗證
   - 工作量證明驗證
   - 交易驗證
   - 狀態更新
```

#### 資源需求

| 項目 | Bitcoin | Ethereum |
|------|---------|----------|
| 存儲需求（2024） | ~500GB | ~1TB+ |
| 增長率 | 每年 100-200GB | 每年 100-200GB |

#### 同步優化
- **並行下載**：多線程區塊下載
- **簽名緩存**：緩存已驗證的交易簽名
- **檢查點**：跳過已知正確的歷史區塊驗證

#### 挑戰與解決方案

| 挑戰 | 解決方案 |
|------|----------|
| 同步時間長 | 更快的硬體 (SSD)、並行處理優化 |
| 存儲空間大 | 區塊修剪 (Pruning)、壓縮存儲 |
| 網路攻擊 | 多樣化節點連接、檢查點驗證 |

#### 同步模式比較

| 特性 | Full Sync | Fast Sync | Light Sync |
|------|-----------|-----------|------------|
| 安全性 | 最高 | 高 | 中等 |
| 同步時間 | 最長 | 中等 | 最短 |
| 存儲需求 | 最大 | 大 | 最小 |
| 驗證程度 | 完全 | 部分 | 最小 |

> **🔄 技術更新（2025+）**：Ethereum 的 **Portal Network** 是一個去中心化的輕客戶端基礎設施，允許節點僅存儲部分歷史數據並彼此共享，從根本上解決全同步的高門檻問題。此外，**Snap Sync**（Geth 預設模式）大幅加快狀態同步速度，已基本取代舊有 Fast Sync。

#### 適用場景
- **全節點運營**：礦工節點、交易所節點、開發節點
- **網路支撐**：維護去中心化、保存完整歷史、為其他節點提供資料
