## 3.1 PoW 難度調整

### Bitcoin 每 2016 塊調整

比特幣的難度調整機制確保平均出塊時間維持在 10 分鐘，維持系統的穩定性和可預測性。

#### 調整公式
```
new_difficulty = old_difficulty × (target_time / actual_time)
```

- **目標時間 (target_time)**：2016 × 10 分鐘 = 1,209,600 秒（兩週）
- **實際時間 (actual_time)**：最近 2016 個區塊的實際產生時間
- **調整上下限**：±4 倍（防止極端波動）

```python
TARGET_TIMESPAN = 14 * 24 * 60 * 60  # 兩週 = 1,209,600 秒
BLOCK_INTERVAL = 10 * 60              # 目標出塊間隔 = 600 秒
ADJUSTMENT_INTERVAL = 2016            # 每 2016 個區塊調整一次

def calculate_new_difficulty(old_difficulty, actual_timespan):
    # 限制調整幅度
    if actual_timespan < TARGET_TIMESPAN // 4:
        actual_timespan = TARGET_TIMESPAN // 4
    if actual_timespan > TARGET_TIMESPAN * 4:
        actual_timespan = TARGET_TIMESPAN * 4
    
    new_difficulty = old_difficulty * TARGET_TIMESPAN // actual_timespan
    return new_difficulty
```

#### 難度目標格式（Compact Target）
難度以壓縮格式（4 bytes）儲存在區塊頭中：
```
compact_target = (exponent << 24) | mantissa
target = mantissa × 256^(exponent - 3)
```

合法區塊必須滿足：`block_hash < target`

#### 算力換算
- **全網算力 (H/s) = 難度 × 2^32 / 出塊時間**
- 2024 年 Bitcoin 全網算力約 500-700 EH/s（Exa = 10^18）

#### 歷史難度演化
```
2009: 難度 = 1 (Satoshi 的筆記型電腦可挖礦)
2013: 難度 = 100,000,000 (ASIC 礦機出現)
2020: 難度 = 15,000,000,000,000 (工業規模挖礦)
2024: 難度 = 80,000,000,000,000+ (算力持續增長)
```

## 3.2 PoS 與以太坊 2.0

### Ethereum 2.0 轉換

以太坊從 PoW 轉換到 PoS 是區塊鏈史上最大規模的協議升級。

#### 時程里程碑
- **2020 年 12 月**：信標鏈 (Beacon Chain) 上線，PoS 開始運行
- **2022 年 9 月**：「合併」(The Merge)，主網切換到 PoS
- **能源削減**：合併後能耗降低 99.95%

#### 信標鏈結構

```
Epoch (6.4 分鐘)
├── Slot 0 (12 秒) → Proposer: Validator_X → Block/Empty
├── Slot 1 (12 秒) → Proposer: Validator_Y → Block/Empty
├── ...
└── Slot 31 (12 秒) → Proposer: Validator_Z → Block/Empty

每 Epoch：
- 32 個 Slot × 12 秒 = 384 秒 ≈ 6.4 分鐘
- 驗證者隨機分配到委員會
- 最終性在 2 個 Epoch 後確認（~12.8 分鐘）
```

#### 驗證者生命週期
```
PENDING (待機)
    ↓ (存入 32 ETH + 等待入場佇列)
ACTIVE (活躍)
    ↓ (志願退出 / 被懲罰)
EXITING (退出中)
    ↓
SLASHED (被罰沒) ← 發現違規行為
    ↓
WITHDRAWN (已提款)
```

**最低質押量**：32 ETH（約 2024 年初 $100,000 USD）

> **🔄 技術更新（2025+）**：
> - **單槽最終性（Single Slot Finality, SSF）**：目前 Ethereum 需要 ~12.8 分鐘（2個 Epoch）達到最終性，SSF 目標是在單一 Slot（12秒）內完成最終性，是 Ethereum 路線圖的重要里程碑，預計 2026+ 實現。
> - **Pectra 升級（2025年）**：包含 EIP-7002（執行層觸發提款）、EIP-7251（提高最大有效餘額至 2048 ETH，允許大驗證者合並），以及帳戶抽象相關的 EIP-7702。
> - **提議者-建設者分離（PBS）**：Ethereum 透過 MEV-Boost 軟體實現 PBS，將區塊提議者和區塊建設者分離，防止驗證者壟斷 MEV 利潤，未來計劃通過 ePBS（協議層 PBS）原生實現。

### RANDAO-VDF 隨機性

隨機數對於 PoS 的驗證者選擇至關重要，RANDAO+VDF 方案確保了高質量的不可預測隨機數。

#### RANDAO 提交-揭示方案

**RANDAO 提交階段**（提前多個 Epoch）
```python
def commit_randao(validator_privkey, epoch):
    # 驗證者生成隨機值並哈希
    random_value = os.urandom(32)
    commitment = sha256(random_value)
    return commitment, random_value  # 保存隨機值，公開 commitment

# 所有驗證者在 epoch 開始前提交 commitment
```

**RANDAO 揭示階段**（輪到出塊時）
```python
def reveal_randao(validator_privkey, epoch, random_value):
    # 揭示之前承諾的隨機值
    reveal = sign(validator_privkey, sha256(epoch.to_bytes(32, 'big')))
    return reveal

def accumulate_randao(current_mix, reveal):
    # 混合所有驗證者的隨機揭示
    return sha256(current_mix + reveal)
```

#### VDF (可驗證延遲函數) 強化

RANDAO 存在最後一個提交者的攻擊向量，VDF 透過引入強制計算延遲解決此問題。

```python
def vdf_sequential_squaring(x, steps, modulus):
    """Pietrzak/Wesolowski VDF: y = x^(2^steps) mod N"""
    for _ in range(steps):
        x = pow(x, 2, modulus)
    return x

class WesolowskiVDF:
    def __init__(self, modulus, delay_seconds=600):  # 10 分鐘延遲
        self.modulus = modulus
        self.steps = delay_seconds * SQUARINGS_PER_SECOND  # 約 10^9 次平方
    
    def evaluate(self, x):
        """計算 y = x^(2^steps) mod N，需要順序執行，無法並行"""
        y = x
        for _ in range(self.steps):
            y = (y * y) % self.modulus
        return y
    
    def prove(self, x, y):
        """生成可高效驗證的 VDF 證明"""
        # Wesolowski 證明：O(T/log T) 步驟
        l = hash_to_prime(x, y, self.steps)  # 隨機質數挑戰
        q, r = divmod(self.steps, l)
        pi = pow(x, q, self.modulus)  # 證明
        return pi
    
    def verify(self, x, y, pi):
        """快速驗證：O(log T) 步驟"""
        l = hash_to_prime(x, y, self.steps)
        r = (2 ** self.steps) % l
        return (pow(pi, l, self.modulus) * pow(x, r, self.modulus)) % self.modulus == y
```

#### 組合架構
```
Epoch N-2: 驗證者提交 RANDAO commitments
Epoch N-1: 驗證者揭示 RANDAO reveals → 累積 RANDAO_MIX
           VDF 計算開始（10 分鐘）
Epoch N:   VDF 輸出就緒 → 最終隨機數 → 選擇委員會和提議者
```

#### Fisher-Yates 洗牌
```python
def compute_shuffled_index(index, index_count, seed):
    """使用隨機數洗牌驗證者列表，分配委員會"""
    for round_num in range(90):  # 90 輪洗牌
        pivot = hash(seed + round_num.to_bytes(1, 'big')) % index_count
        flip = (pivot + index_count - index) % index_count
        position = max(index, flip)
        source = hash(seed + round_num.to_bytes(1, 'big') + (position // 256).to_bytes(4, 'big'))
        byte = source[(position % 256) // 8]
        bit = (byte >> (position % 8)) & 1
        index = flip if bit else index
    return index
```

## 3.4 先進共識機制

### 拜占庭容錯算法

BFT 算法能在存在惡意節點的分散式系統中達成一致，廣泛用於許可鏈和 PoS 系統的最終性確認。

#### PBFT（實用拜占庭容錯）

**三階段協議**
```
PRE-PREPARE: 主節點廣播提案
    │
PREPARE: 節點確認提案並廣播（需收到 2f+1 個 PREPARE）
    │
COMMIT: 節點確認準備狀態並廣播（需收到 2f+1 個 COMMIT）
    │
REPLY: 向客戶端返回結果
```

**容錯能力**
- 最多 f 個惡意節點：`f = (n-1) // 3`
- 最少需要 `3f+1` 個節點（4 個節點可容 1 個惡意節點）

```python
class PBFTNode:
    def __init__(self, node_id, total_nodes):
        self.node_id = node_id
        self.n = total_nodes
        self.f = (total_nodes - 1) // 3
        self.view = 0
        self.sequence = 0
        self.prepare_count = {}
        self.commit_count = {}
    
    def is_primary(self):
        return self.node_id == self.view % self.n
    
    def send_pre_prepare(self, request):
        if not self.is_primary():
            return
        self.sequence += 1
        msg = {'type': 'PRE-PREPARE', 'view': self.view, 
               'seq': self.sequence, 'request': request,
               'signature': self.sign(request)}
        self.broadcast(msg)
    
    def on_receive_prepare(self, msg):
        key = (msg['view'], msg['seq'])
        if key not in self.prepare_count:
            self.prepare_count[key] = set()
        self.prepare_count[key].add(msg['sender'])
        
        # 收到 2f+1 個 PREPARE 消息（包含自己）
        if len(self.prepare_count[key]) >= 2 * self.f + 1:
            self.send_commit(msg)
    
    def on_receive_commit(self, msg):
        key = (msg['view'], msg['seq'])
        if key not in self.commit_count:
            self.commit_count[key] = set()
        self.commit_count[key].add(msg['sender'])
        
        # 收到 2f+1 個 COMMIT 消息
        if len(self.commit_count[key]) >= 2 * self.f + 1:
            self.execute_request(msg['seq'])
```

**視圖切換（主節點故障恢復）**
```python
def initiate_view_change(self):
    """主節點超時，發起視圖切換"""
    new_view = self.view + 1
    msg = {'type': 'VIEW-CHANGE', 'new_view': new_view,
           'prepared': self.get_prepared_requests()}
    self.broadcast(msg)

def on_receive_view_change(self, msg):
    new_view = msg['new_view']
    if new_view not in self.view_change_count:
        self.view_change_count[new_view] = set()
    self.view_change_count[new_view].add(msg['sender'])
    
    # 收到 2f+1 個 VIEW-CHANGE 消息
    if len(self.view_change_count[new_view]) >= 2 * self.f + 1:
        self.view = new_view
        if self.is_primary():
            self.send_new_view(new_view)
```

#### HotStuff（線性 BFT）

HotStuff 是 PBFT 的改進版，實現線性通信複雜度，由 Facebook 的 LibraBFT/DiemBFT 採用。

**核心改進**
- **線性訊息複雜度**：O(n) 而非 PBFT 的 O(n²)
- **門檻簽名**：聚合多個驗證者簽名為單一的 Quorum Certificate (QC)
- **管道化**：多個 Epoch 可以並行進行（三相流水線）

```python
class HotStuffNode:
    def __init__(self, node_id, n):
        self.node_id = node_id
        self.n = n
        self.f = (n - 1) // 3
        self.quorum_size = 2 * self.f + 1
        self.current_view = 0
        self.locked_qc = None  # 已鎖定的 Quorum Certificate
        self.high_qc = None    # 見過最高的 QC
    
    def on_receive_proposal(self, proposal):
        block = proposal['block']
        qc = proposal['qc']
        
        # 安全條件：新提案的 QC 比 locked_qc 更高
        if self.extends(block, qc.block) or qc.view_number >= self.locked_qc.view_number:
            self.vote(block)
    
    def create_qc(self, votes):
        """聚合簽名形成 QC"""
        if len(votes) >= self.quorum_size:
            aggregated_sig = self.aggregate_signatures([v.signature for v in votes])
            return QuorumCertificate(
                block_hash=votes[0].block_hash,
                view_number=self.current_view,
                signature=aggregated_sig
            )
    
    def three_phase_commit(self, block):
        """三階段提交：PREPARE → PRE-COMMIT → COMMIT"""
        # PREPARE 階段
        prepare_qc = self.collect_votes('PREPARE', block)
        # PRE-COMMIT 階段  
        self.locked_qc = prepare_qc  # 鎖定（防止長程攻擊）
        precommit_qc = self.collect_votes('PRE-COMMIT', block)
        # COMMIT 階段
        commit_qc = self.collect_votes('COMMIT', block)
        self.execute_block(block)
        return commit_qc
```

#### PBFT vs HotStuff 比較

| 特性 | PBFT | HotStuff |
|------|------|---------|
| 通信複雜度 | O(n²) | O(n) |
| 延遲（輪次） | 2-3 輪 | 3 輪 |
| 視圖切換 | 複雜 O(n²) | 簡單 O(n) |
| 簽名方案 | 個別驗證 | 門檻/聚合簽名 |
| 適用規模 | ≤100 節點 | 數百-數千節點 |

## 3.5 攻擊防範

### 51% 算力攻擊

51% 攻擊是指攻擊者控制超過全網算力的 51%，能夠重新組織區塊鏈歷史。

#### 攻擊類型

**雙重花費攻擊**
```
1. 攻擊者向受害者發送交易（公鏈）
2. 同時私下挖掘包含不同交易的替代鏈
3. 等待受害者確認交易（通常 6 個確認）
4. 公布比公鏈更長的替代鏈
5. 公鏈被孤立，原交易被撤銷
```

**自私挖礦 (Selfish Mining)**

Eyal & Sirer (2013) 分析：
```python
def selfish_mining_revenue(alpha):
    """alpha = 攻擊者算力比例"""
    # 攻擊者收益高於誠實比例的臨界值
    if alpha >= 1/3:  # alpha > 0.333
        gamma = 0.5  # 攻擊者連接優勢
        revenue = (alpha * (1 - alpha)**2 * (4*alpha + gamma*(1 - 2*alpha)) - alpha**3) / \
                  (1 - alpha*(1 + alpha*(2 - alpha)))
        return revenue
```

當算力超過 ~33%，自私挖礦可獲得高於算力比例的收益。

**審查攻擊**
- 攻擊者拒絕包含特定交易的區塊
- 挖礦只在自己挖出的空塊後繼續

#### 攻擊成本分析

```python
def attack_cost_calculator(hashrate_fraction, network_hashrate_ehs, electricity_cost, hardware_cost_per_th):
    """計算 51% 攻擊成本"""
    attack_hashrate = hashrate_fraction * network_hashrate_ehs * 1e18  # H/s
    
    # 硬體成本（一次性）
    hardware_th = attack_hashrate / 1e12  # TH/s
    total_hardware = hardware_th * hardware_cost_per_th
    
    # 電力成本（每小時）
    power_consumption_kw = hardware_th * 0.03  # 假設 30W/TH
    hourly_electricity = power_consumption_kw * electricity_cost / 1000
    
    return {
        'hardware_cost': total_hardware,
        'hourly_electricity': hourly_electricity,
        '1h_attack_cost': total_hardware + hourly_electricity
    }

# Bitcoin 2024 年約需 $10-30 億美元 + 每小時數百萬電費
```

### 長程攻擊防範

長程攻擊是 PoS 系統特有的威脅，攻擊者使用歷史私鑰重寫早期區塊鏈歷史。

#### 無利害關係問題 (Nothing-at-Stake)

**問題描述**
```
PoW：礦工在分叉時必須選擇，算力有限
PoS：驗證者在分叉時可以同時在兩條鏈上簽名，無額外成本
```

#### 長程攻擊向量

#### 弱主觀性 (Weak Subjectivity)

Vitalik Buterin 提出的解決方案：

```python
WEAK_SUBJECTIVITY_PERIOD = 2048  # Epochs（約 9 天）

def verify_chain_with_weak_subjectivity(chain, checkpoint):
    """
    新節點加入時，需要一個足夠新的可信檢查點
    從檢查點開始同步，而非從創世區塊
    """
    if chain.contains_checkpoint(checkpoint):
        if checkpoint.age_in_epochs() <= WEAK_SUBJECTIVITY_PERIOD:
            return True
    return False  # 拒絕太舊的鏈
```

**相關性懲罰**：若大量驗證者同時被懲罰，個別懲罰額度大幅提高，防止協調攻擊。

#### Casper FFG 最終性
```
每 2 個 Epoch（約 12.8 分鐘）達到最終性
最終化後的區塊永不可逆（否則需燒毀 1/3 ETH 質押）
```

```python
class CasperFFG:
    def __init__(self):
        self.justification = {}  # 已驗證但未最終化
        self.finalization = {}   # 已最終化
    
    def process_vote(self, source_epoch, target_epoch, validator_id):
        """Casper FFG 投票"""
        if target_epoch not in self.votes:
            self.votes[target_epoch] = set()
        self.votes[target_epoch].add(validator_id)
        
        # 超過 2/3 驗證者投票 → Justify
        if len(self.votes[target_epoch]) > 2/3 * self.total_validators:
            self.justify(target_epoch)
    
    def justify(self, epoch):
        self.justification[epoch] = True
        # 連續兩個 Justified epoch → Finalize 第一個
        if epoch > 0 and self.justification.get(epoch-1):
            self.finalize(epoch-1)
```
